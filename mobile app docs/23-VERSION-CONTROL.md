## 3. State visibility: "where are we at", at any moment

**(a) Schema.** The rule (also in the repo README and project CLAUDE.md): **the `supabase/migrations/` folder IS the schema. If it is not in a migration, it does not exist.** Check anytime: `supabase migration list --linked` against dev or prod (any row present on one side only is drift). CI enforces it twice (types-drift diff + full-history apply on a fresh DB per PR). **Studio hotfix exception:** allowed only for the owner in a genuine emergency; within 24h the same SQL lands as a numbered migration in a PR (marked applied via `migration repair`) with a one-line why. Any other remote-only row in `migration list` is an incident.

**(b) Releases.** Annotated git tag per store submission (`v1.2.0`, triggers release.yml); EAS records the git commit on every build and OTA update; `docs/runbooks/releases.md` keeps a small table: tag, versionCode/buildNumber, runtime fingerprint, rollout %. That table + the store consoles is the whole release picture.

**(c) Work.** GitHub Issues + **Milestones mapped 1:1 to build phases** (Phase -1, 0, 1, 2, 3, 4) + ONE pinned GitHub Project board. No hand-edited PROGRESS.md (goes stale the first busy week; the board auto-updates from issue/PR state and gives church stakeholders a readable URL). The launch checklists in docs 18/19/22/24 are converted into checklist issues so every checkbox is trackable. "Where are we" = the board; "what remains for launch" = open items in the current milestone.

**(d) Decisions.** ADRs in `docs/decisions/NNNN-title.md` (minimal MADR); the ten from `21` §1 are backfilled on day one. Any decision that changes architecture, a vendor, or money gets an ADR in the same PR that implements it. Superseded ADRs are marked, never deleted.
