Every screen, every action, every destination

This is the blueprint — not the build. Each card is a screen with an ID. Under it, every button/action shows where it goes (a bold ID is a jump to that screen). Any action that needs sign-in is marked gate — guests get routed to Auth first, then bounced back. Nothing dead-ends.

Global rules — how we guarantee no dead ends
Back everywhere. Every non-tab screen has a ↩ back/close to its parent. Tab roots can’t go “back” — the tab bar is always the escape.
Gated actions never fail silently. A guest tapping a gate action always lands on A1, and on success returns to exactly where they were.
Every list has an empty state. No blank screens — each shows a prompt + a next action (see SYS).
Offline / error / loading. Global banners + retry; skeletons while loading; a toast confirms every save/reaction.
External hops return. Web checkout & book buy open a browser and route back to an in-app confirmation.
Push deep-links. Every notification opens the exact relevant screen, not just Home (see SYS·PN).

Added — closing the last gaps

The five items that weren’t explicitly mapped. With these in, v1 is complete. For the two that are behaviour (not just a screen), the recommended method is called out.
NC1Notification centerBoth
Bell in the H1 header. In-app list of past pushes; tap an item → its deep-link (W2/FD1/FD2/EV2). Empty state + mark-all-read. ↩ H1.
F10Edit / delete my postMember
From F9: Edit → re-opens FC1/FC2 prefilled (edits re-enter review). Delete → confirm → removed + toast → F9. (App-store requirement: users can remove their content.)
E1bPrivacy & consentBoth
Layered notice, NOT a blocking screen (docs 06/20): a short privacy line on E1 + full policy at LG1. Explicit consent is captured contextually at compose (FC1/FC2) where the Art. 9 sharing actually happens. Policy changes surface as a dismissible notice.
✦ #3Session expiry — method
Silent refresh. Short-lived access token (~1h) + long-lived rotating refresh token (~60d), renewed in the background. The user never sees a login unless the refresh token itself fails/revoked — only then → A1, returning to the exact screen after. Signing out or ACC5 revokes it.
✦ #5Account recovery — method
v1: lost number = new profile (documented limitation, docs 03). An admin can merge on request only after OTP-verifying control of both numbers (account-takeover guard, docs 20). Recovery email / passkeys are post-v1 options.

v1 ships everything except the phase 2 items (nation detail F6, series, groups, giving history). The Nations map F5 itself is v1: one third of the wedge (docs 09). On any conflict between this map and the docs, the docs win (see AUDIT-2026-07-12.md Part D).
