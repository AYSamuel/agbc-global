# Production Readiness Audit (2026-07-12)

Five independent adversarial reviews (data model and migrations; mobile implementation currency; DevOps, QA and release; security and abuse; product, UX and content operations), each judged against the standards library and with time-sensitive claims verified against current official sources (Expo changelogs, Supabase docs, Twilio, Meta, Google Play, Payhip, YouTube ToS). Findings are deduplicated and ordered by severity. Settled decisions (Expo + RN, Supabase, phone OTP WhatsApp-first, moderation-first, replace Grace Portal, reuse shared Supabase project) were not relitigated.

**Verdict:** the product spec (flows, screens, data intent, compliance intent) is strong. What is not yet solid is the layer a production company lives on: server-side write enforcement, operational runbooks, testing, content operations, and several technology choices that 2026 has invalidated since the docs were written. Everything below is fixable in documentation now; most of it is expensive to retrofit after Phase 0.

---

## A. Blockers (fix in docs before Phase 0)

### A1. Server-side write invariants do not exist; moderation is bypassable three ways
The docs say "never trust the client" but never pin down the column-level rules that make it true. With the anon key + a member JWT, `supabase-js` can set ANY column on insert/update unless constrained:
- Insert a testimony/prayer with `status='approved'`: skips moderation entirely.
- Update own `profiles.role` to `admin`: instant global power (role immutability is stated in 03 but not enforced anywhere).
- Edit an APPROVED post's body/photo: republishes unmoderated content (09 allows owner edit; nothing resets status).

**Fix:** new "RLS and write-path invariants" section in `02`: per table, which columns each role may set or change. Enforced by BEFORE INSERT/UPDATE triggers (force `status='pending'`, `author_id=auth.uid()`, `branch_id`=author's branch; raise if a non-admin changes `role`; any author edit to body/image/category on an approved row resets `status='pending'` and clears moderation fields) plus RLS WITH CHECK. Add CI authorization tests that attempt each bypass and assert failure. Also add `rejection_reason text` to both content tables (see C7).

### A2. The dashboard's service-role usage makes route code the only authorization layer
Service role bypasses RLS completely. One dashboard route that trusts a request-supplied `branch_id`, or forgets a role check, breaks all tenant isolation. **Fix (17):** centralized authz middleware on every server route: derive caller identity from their session JWT, load role+branch server-side, authorize action+target before any service-role call; prefer the caller's own JWT + RLS where possible; IDOR tests with foreign branch ids in CI.

### A3. Storage buckets are unspecified; pending testimony photos would be publicly fetchable
A photo on a PENDING testimony must not be reachable before review (moderation bypass for images, CSAM exposure window). **Fix (02/09/14/16):** enumerate buckets and policies: `testimony-photos` private, signed URLs minted only for approved rows; `book-files` private, signed per-request after entitlement check, short TTL; `avatars` public-read acceptable; `sermon-audio` public or modest-TTL signed. All uploads: authenticated, size-capped, magic-byte validated, images re-encoded with EXIF/GPS stripped (testimony photos can carry a member's home GPS), random ids.

### A4. No timezone exists anywhere in the schema, yet three features depend on branch-local time
`attendance.service_date` "computed in the branch's timezone" (10), service reminders "per branch service time/timezone" (15), events "in branch timezone" (11): but `branches` has no timezone column and `service_times` is display-string jsonb ("Mittwochs 19:00 Uhr") no scheduler can parse. UK/DE/NL observe DST; Lagos does not; fixed-UTC schedules drift twice a year. **Fix (02):** `branches.timezone` (IANA); new `branch_services` table (branch_id, weekday, start_time, kind, label) as the machine-readable schedule, jsonb kept for display; `events` store wall-clock + zone id per the backend standard; define `service_date = (now() at time zone branches.timezone)::date`; state which timezone anchors `daily_verses.date`.

### A5. GDPR Art. 9 consent evidence has no home in the schema
Doc 20 rests the lawful basis on "consent stored with timestamp and wording version"; 02 has nowhere to store it. **Fix (02):** `consent_version text not null` + `consented_at timestamptz not null` on `testimonies` and `prayers`; versioned wording file in the repo; 16 states what happens to consent records on deletion (retain as processing evidence).

### A6. Realtime as specced leaks anonymous prayer authors and cannot use the protective view
Verified against current Supabase docs: `postgres_changes` streams raw base-table rows (publications cannot include views), RLS filters rows not columns, DELETE events are not RLS-filtered, and Supabase now recommends Broadcast for scalability and security. The view that strips `author_id` protects REST reads only; the live feed would deliver `author_id` on every anonymous prayer. **Fix (01/02/09):** replace `postgres_changes` with broadcast-from-database: AFTER INSERT/UPDATE triggers call `realtime.broadcast_changes()` on private channels (`family:branch:<id>`, `family:all`) with a sanitized payload (author fields nulled when anonymous; only `approved` transitions broadcast); RLS policy on `realtime.messages`; Realtime Authorization on, public access off.

### A7. The mobile stack has three 2026-stale choices (verified against Expo changelogs)
- **expo-av was removed in Expo SDK 55** (current stable: SDK 57). Its replacement `expo-audio` now covers exactly what doc 08 needs: background playback via config plugin, lock-screen/now-playing controls (`setActiveForLockScreen`), playlists, speed. `react-native-track-player` is the wrong primary bet (v4 unreliable under the now-mandatory New Architecture; v5 is `@rntp/player` under a commercial license). **Fix (01/08):** expo-audio is the audio engine; RNTP demoted to an evaluated fallback.
- **Expo Go is frozen at SDK 54 in the stores** and Expo now directs production teams to development builds; push notifications already broke in Go on Android since SDK 53. The "Expo Go covers ~90% of iOS testing from Windows" workflow is obsolete. **Fix (01 §2, 18 Phase 0):** the standing workflow is an EAS development build (expo-dev-client) installed on the real iPhone (register device, ad hoc distribution); hot reload from Windows works the same; rebuild only when native deps change.
- **Supabase sessions exceed expo-secure-store's 2048-byte limit.** **Fix (03):** `auth.storage` = the Supabase-recommended LargeSecureStore adapter (AES key in SecureStore, ciphertext in AsyncStorage).

### A8. In-app user blocking is a store requirement for UGC apps and is deferred post-v1
Apple 1.2 and Play's UGC policy require block, not just report. Discovering this at Phase 4 submission is the worst timing. **Fix (02/03/04/09):** v1 ships `blocked_users` (blocker_id, blocked_id), "Block this member" in the card overflow, feeds filter blocked authors server-side, Settings "Blocked members" list.

### A9. The prototypes contradict the docs on legally load-bearing flows
AppFull has NO consent step and its prayer submit copy says content is "shared right away" (unmoderated); its onboarding includes a blocking privacy screen and a notifications step the docs forbid; the Screen Map tags the global family map "ph2" (descoping one third of the v1 wedge), shows Google/Apple OAuth, and specs an account-recovery flow 03 explicitly rejects. A developer "building to match" ships a GDPR breach and an unmoderated prayer wall. **Fix (README/18):** add a "Prototype deltas" section: docs win on behavior; enumerate the full delta list (Part D below). Optionally patch AppFull's compose + onboarding, which teach the wrong thing on every viewing.

### A10. Migrations into the shared prod project are ungated and unrehearsed
No environment resembles prod (website tables exist only there) until the moment prod is touched; nothing gates who can `db push`. **Fix (19):** local `supabase start` is the daily dev loop; after cleanup, `supabase db pull` produces a baseline migration INCLUDING the website tables (dev then truly matches prod) with `supabase migration repair`; a one-time rehearsal: dump prod, restore into a scratch project, run the entire Grace Portal drop + new schema end to end; prod deploys only via a GitHub Actions job bound to a `production` environment with required-reviewer approval; a pgTAP "fence guard" test asserts the website objects are untouched; the fence audit must cover GRANTs too (one shared anon key means every anon policy on website tables is reachable from the app).

### A11. Production must be Supabase Pro; the docs' backup assumptions are wrong
Verified: Free tier has NO automatic backups; Pro has daily backups with 7-day retention (doc 20 says "~30 days": wrong); PITR is a paid add-on; Storage objects (sermon audio, book files, avatars) are NOT in database backups at all; free projects pause after ~7 days idle; Free egress (5GB/month) is ~100 total sermon-audio listens. **Fix (18/19/20):** prod on Pro before any cleanup; correct 20's retention row; nightly `supabase db dump` off-provider via GitHub Actions plus a Storage backup job; restore drill on the launch checklist; note Realtime concurrent-connection ceilings (Free 200 / Pro 500) against the Sunday-live peak.

### A12. There is no testing strategy anywhere
**Fix (new 21-OPERATIONS.md):** the concrete pyramid: Jest (`jest-expo`) for pure logic (streak/ISO-week math with DST cases, gate-return, offline queue); React Native Testing Library for the four data states, GateSheet, consent flow; pgTAP via `supabase test db` in CI for RLS (guest sees only approved; anonymous never leaks author_id; the A1 bypass attempts); `deno test` for edge functions incl. Payhip replay fixtures; zod contracts in a shared package + generated DB types diffed in CI; Maestro for E2E journeys (the 2026 Expo consensus over Detox); device matrix: one low-end real Android on the minimum OS + current iPhone at small/large/tablet widths and 2x text scale. Per-PR vs nightly cadence defined.

### A13. Background jobs have no operational spec
YouTube sync, streak recompute, service reminders, broadcast fan-out, Payhip processing, token pruning, retention purges: no scheduler, retries, idempotency, or failure alerting named. Verified: Supabase Cron (pg_cron + pg_net) is the scheduler but does not retry skipped runs or alert. **Fix (21-OPERATIONS.md):** per-job table (schedule, idempotency key, retry, blast radius); broadcast fan-out chunked through `broadcast_deliveries` rows; Expo push is two-phase (tickets then receipts ~15 min later; prune tokens on `DeviceNotRegistered` from RECEIPTS, or delivery decays and Expo throttles you); every job pings a healthchecks.io dead-man check so silence itself alerts.

### A14. Content that must exist before code does
- **Devotional plan:** a v1 surface with zero content and no named author. A 30-day plan is 9,000 to 15,000 words of pastoral writing; commission it NOW (owner + template + deadline end of Phase 2). Until approved, Home's CTA must not route to an empty PLAN.
- **Daily verses:** 365+ rows/year forever, no owner, and the dashboard entry UI is scheduled three phases after the feature ships. Move verse CRUD to dashboard Phase A; owner + quarterly batch of 90; edge-function alert when fewer than 14 future days are queued; seed 90 days pre-launch.
- **Branch seed data is missing the map:** verified against `agbc/src/content/branches/*.json`: no lat/lng, no slug, no timezone; keys are camelCase (`isHq`, `youtubeChannelId`, `times` not `service_times`). Seeding "from these files" yields a flagship map with ZERO pins. Fix: versioned seed file merging the JSON with a hand-built augmentation map (geocoded lat/lng, timezone, slug).
- **Family cold start:** the wedge launches as an empty feed and a fictional map. Fix: founding-members program (15-25 per branch on the beta track, each contributing one testimony + one prayer), pre-launch collection at services, leaders post first, launch-Sunday choreography; bar: 10+ approved testimonies across all 4 branches before public launch.

## B. High severity

**B1. Payhip webhook is forgeable.** Verified: Payhip's "signature" is `hash('sha256', apiKey)`: a constant, no payload HMAC, no timestamp. Anyone who ever sees one webhook can mint entitlements. Fix (14): treat the webhook as a trigger only; confirm the sale via Payhip's server API (license verify / transactions) before granting; store the transaction id with `unique(source_ref)` for idempotent replays; `unmatched_purchases` table drained when a profile later verifies that email; refund events revoke.

**B2. Restore-purchase claim abuse.** Typed emails are attacker-controlled (claim someone else's purchase; enumeration oracle). Fix (14): grant only against the profile's VERIFIED email or require the Payhip order id; uniform responses; rate-limited.

**B3. Leader/admin plane is phone-OTP only.** SIM-swap + OTP phishing vs accounts that can broadcast to the whole ministry and read reports. The security standard requires phishing-resistant MFA for staff. Fix (17): passkeys on the dashboard, mandatory for admins; step-up re-auth for role changes and ministry broadcasts; dedicated admin accounts; immutable audit log.

**B4. Broadcast blast radius.** No confirmation, caps, or second pair of eyes. Fix (15/17): confirmation screen with exact recipient count and rendered link; link allowlist/preview; per-account daily caps; four-eyes approval for ministry scope; a halt control for in-flight fan-out.

**B5. OTP toll fraud (SMS pumping).** Nigeria corridor is a known target. Fix (03): Twilio Verify Fraud Guard stays on; Verify Geo Permissions allowlist of served countries; layered per-number/IP/device limits; Twilio spend alerts + cap; same limits on the SMS fallback path; optionally reject VoIP line types via Lookup.

**B6. Review OTP bypass hardening.** As written it is a permanent fixed-code account. Fix (03): feature-flagged off outside review windows; exactly one allowlisted number; long random code; account pinned to member role and cannot post publicly; alert on every use.

**B7. CSAM/illegal-image duty.** User photo uploads with a global audience. Fix (20/17): hash-matching (Cloudflare CSAM tool if fronted by CF, else PhotoDNA/Thorn), a documented takedown + report path (NCMEC/IWF/police), moderators trained never to download flagged content; pending photos already private per A3.

**B8. WhatsApp cost model is an order of magnitude stale.** Verified: Meta moved to per-message billing; a ministry-wide marketing-category blast to 2,000 opted-in members is roughly $200 PER BLAST at UK/DE rates. Fix (01/15): correct the cost section; broadcasts default to push + in-app; WhatsApp channel rationed (suggested: max 2 ministry-wide WhatsApp blasts/month) with template categories chosen deliberately.

**B9. CI/CD is one line.** Fix (21-OPERATIONS.md): per-PR path-filtered checks; EAS profiles development/preview/production mapped 1:1 to EAS Update channels; secrets map (GitHub repo vs `production` environment vs EAS env vs Supabase function secrets vs Vercel scopes); protected main; Renovate with security fast-lane + cooldown.

**B10. Observability: nothing alerts.** Fix (21): the minimal owned alert set: Sentry (app, dashboard, edge functions) with rollout halt threshold; healthchecks.io dead-man pings; OTP success-by-country funnel (alert if NG < 70%/24h, plus Twilio error/spend triggers); broadcast failure >10%; UptimeRobot on the giving page, website, dashboard, REST endpoint; Supabase usage at 80%.

**B11. Bus factor 1.** Fix (21 + runbooks): keystore copies (EAS + church password-manager vault + offline in the church safe), credential inventory with second owners for Apple/Google/Supabase/Twilio/Meta/Payhip/EAS/GitHub, second Supabase org owner, billing alerts, deploy/rollback/restore/rotate runbooks.

**B12. Repo structure undecided.** Fix (01/21): pnpm monorepo: `apps/mobile`, `apps/dashboard`, `packages/shared` (zod contracts + generated DB types), `supabase/` (migrations, functions, tests, seed), `docs/spec` (this folder moves in), `docs/decisions` (ADRs: reuse-shared-supabase, replace-grace-portal, whatsapp-first-otp, link-out giving, expo-audio, Maestro), `docs/runbooks`.

**B13. Release engineering.** Fix (01/18/21): `runtimeVersion: { policy: "fingerprint" }`; OTA = JS/asset fixes only, percentage rollout, never the forced-update mechanism; staged rollout 10>25>50>100 with written halt criteria (crash-free <99.5%, ANR >0.5%, OTP collapse, moderation/data bug); forced update = Supabase-hosted min-version check + `expo-in-app-updates` on Android + store link on iOS; ship on Expo SDK 56+ (Play requires target API 36 for all updates from 2026-08-31, which lands mid-build); expect a full-length store review for the rebrand release.

**B14. The reader is a hidden mini-project.** EPUB stack is one-maintainer and needs a file-system-API compatibility patch; react-native-pdf has open New-Architecture issues on iOS. Fix (14/18): name the stack (@epubjs-react-native/core + webview; @kishannareshpal/expo-pdf first for PDF), size it honestly (2-3 weeks), and offer the cut: v1 supports the ONE format the actual catalog uses.

**B15. Deletion reach is unenumerated.** 15+ tables untouched by the current spec (reactions, attendance, rsvps, notes, entitlements, reports.reporter_id, devices...); counters drift when reaction rows vanish without trigger-firing paths; pending posts sitting in the moderation queue after account deletion could be approved AFTER consent is withdrawn (a genuine Art. 9 problem). Fix (16/02): a deletion-reach table (per table: hard delete / anonymize / retain, order of operations); pending items hard-cancelled and never approvable; Storage objects deleted in the same job; nightly counter reconciliation.

**B16. RLS needs a matrix and the performant pattern.** Per-row subqueries against `profiles` are the known RLS footgun. Fix (02): per-table policy matrix; role + branch_id in JWT claims via the Custom Access Token hook (server-set), with the propagation caveat (demoted leader keeps claims until refresh, so moderation actions re-check the table); `(select auth.uid())` wrapping; FORCE RLS; `(branch_id, status, created_at desc)` composite indexes on feed tables.

**B17. Sermons sync has no dedupe key.** Fix (02/08): partial unique index on `sermons(youtube_id)` where not null; sync upserts with explicit conflict target; job lock + retry documented.

**B18. Circular FK in the prayer-testimony loop.** Two nullable FKs will drift (A points at B, B points at C). Fix (02): single source of truth: `testimonies.from_prayer_id` UNIQUE with `on delete set null`; drop `prayers.testimony_id`; derive the reverse by join.

**B19. Branch context model is ambiguous where the multi-branch promise lives.** Berlin member switches the chip to Glasgow and taps "I'm here": which branch_id? Whose timezone computes the streak week? Which reminders fire? 07 vs 09 also disagree on whether the chip changes the Family scope default. Fix (07 + 10/15/16): browsing context (chip) is view-only; home branch (profile) drives attendance timezone, reminders, and branch-tier notifications; "I'm here" while browsing another branch records attendance AT the browsed branch (real visits happen) and streaks count any branch; the sheet gains an explicit "Make this my home branch" action; the chip never changes the Family scope default.

**B20. Notifications and broadcasts have no language model.** Automated notifications baked as English strings break the EN/DE/NL promise on the lock screen, the app's most visible surface. Fix (02/15/17): `notifications` store a template key + params jsonb, rendered per recipient `profiles.language` at send and display time; broadcasts get optional per-locale bodies with a fallback and a visible "sent as written to all recipients" notice; name translation reviewers for urgent ministry-wide sends.

**B21. Moderation rejection UX unspecified.** Fix (09/02): MY-POSTS status chips (Pending / Approved / Needs changes / Removed); rejection reason shown inline with "Edit and resubmit" returning to pending; any edit re-pends (ties to A1).

**B22. Event cancellation does not exist.** Churches cancel and move events constantly; deleting a published event strands RSVPs and calendar entries silently. Fix (02/11/17): `events.status (scheduled|cancelled)`; cancelling (never hard-deleting with RSVPs) auto-notifies non-cancelled RSVPs, same on time/venue change; cancelled treatment on EVENT-DETAIL.

**B23. First launch offline dead-ends onboarding.** ONB-2 needs branches from the backend with no cache yet. Fix (06): bundle a read-only branch snapshot (same seed data) as fallback; refresh on connectivity; same for one evergreen verse.

**B24. Store product completeness.** Age-rating questionnaires (Apple's new tiers, Play IARC) must match the UGC + 16+ answers; DE/NL store listings and screenshot matrix need owners; Grace Portal installs wake up to a different app and need release-note copy explaining the change and the new sign-in. Fix (19): a "Store product" section with the answer sheet, localization owners, screenshot matrix, and release-note copy.

**B25. Bible text licensing is unaddressed.** Storing and redistributing a modern translation (NIV/ESV) in your DB, plus branded share images, is licensed use. Fix (07/20): pick the translation (public domain WEB/KJV for zero friction, or a licensed one within its quotation policy with the attribution line rendered on VerseCard, PLAN-DAY, and share images); add a `translation` column.

## C. Medium and low (condensed)
